# GT Studio 邏輯指南（重建版）

> 本文件說明目前系統的核心流程與模組責任，讓新加入開發者可以快速理解「為什麼這樣設計」。

## 一、系統目標

GT Studio Web 是一個內部管理網站，整合以下日常流程：
- Google 登入與權限判斷
- 收據上傳、審核與查詢
- 排程（Schedule）管理
- 權限／成員管理
- Tickets 管理

## 二、技術架構（目前）

- 前端：原生 JavaScript（ES Module）
- 後端服務：Firebase（Auth / Firestore / Storage）
- 入口檔：`app.js`
- 功能模組：`js/modules/*`

### 為什麼用這個架構
- 不引入框架可降低重構成本，先把單體邏輯拆分成可維護模組。
- 入口檔只做初始化，功能邏輯集中在模組，便於多人協作與除錯。

## 三、啟動流程（Bootstrap）

1. `app.js` 載入 Firebase 設定
2. 呼叫 `initApp()` 啟動模組協調層
3. 啟動成功後隱藏 loading
4. 若啟動失敗，顯示錯誤提示

> 設計意圖：
> 把「啟動失敗處理」留在入口層，避免失敗時畫面無回饋。

## 四、主要資料流

### 1) 認證流程
- 使用者按下 Google 登入
- Firebase Auth 完成登入
- 依登入狀態切換導覽與頁面可見內容

### 2) 收據流程
- 使用者填寫表單並上傳附件（Storage）
- 建立收據文件（Firestore）
- 管理員在清單審核（approved/rejected）

### 3) 排程流程
- 讀取當月事件
- 建立／更新行程
- 依日期渲染行事曆與詳情 modal

### 4) 成員與權限流程
- 管理員開啟成員管理
- 儲存角色設定
- 重新渲染角色與成員清單

## 五、模組責任（目前共識）

- `auth`：登入/登出與使用者狀態
- `navigation`：頁面切換與導覽狀態
- `receipts`：收據上傳、清單、審核
- `schedule`：行事曆與排程互動
- `admin`：成員與權限設定
- `tickets`：票務資料流程

> 設計意圖：
> 每個模組只處理單一領域，避免功能交錯造成修改風險。

## 六、目前可維護性風險

1. 專案仍保留 `app_backup.js`、`temp_app.js`，容易造成「哪個才是正式版本」的混淆。
2. 部分共用狀態仍可能透過全域物件間接存取，模組邊界尚未完全收斂。
3. 文件與實作偶有不同步，新增功能前須先確認現況。

## 七、下一步重構建議（短期）

1. 確立唯一正式程式碼路徑（`app.js` + `js/modules/*`）
2. 將備份檔移到 `docs/archive/` 並加註唯讀
3. 收斂模組依賴介面（集中注入 context）
4. 補齊每個模組的公開 API 說明

## 八、維護規範（建議遵守）

- 命名需可讀，避免縮寫
- 每個函數只做一件事
- 關鍵邏輯加上繁體中文註解（說明為什麼）
- 大改前先更新 `progress.md`
